Original PFoodReq Evaluation Pipeline
Step 1: Candidate Generation (via KG traversal)
For each query, PFoodReq uses the topicKey (tag URI) to traverse FoodKG:


tag node ("moose") → [tagged_dishes edge] → ALL recipes under that tag
For Example 0, this gives all 46 moose recipes. These are the origin_answers in the data — recipes that match the original query's basic constraint (tag + excluded ingredient from qOriginText).

Step 2: Constraint Filtering (what BAMnet learns to do)
The full query (qText) adds persona + nutrition + guidelines on top. BAMnet learns to score each candidate against ALL constraints:

Constraint source	Example 0
qOriginText	moose tag, no "ground deer meat"
explicit_nutrition	low carbohydrates (0-45g)
persona.ingredient_likes	must have "dry red wine"
persona.ingredient_dislikes	must NOT have "tomato paste", "sugar"
guideline	sugar 10-35g per meal
BAMnet encodes these constraints into a query vector, scores all 46 candidates, and outputs the ones that pass. The answers field (["Venison Marinade", "Moose Stew"]) are the ground truth — the only 2 out of 46 that satisfy every constraint.

Step 3: Evaluation Metrics
They compare predicted recipe names against answers:

MAP (Mean Average Precision) — considers rank order of correct predictions
MAR (Mean Average Recall) — considers how many correct answers were found
F1 — harmonic mean of precision and recall per query, macro-averaged
